费曼有一个习惯：把想弄懂的概念写下来，用最简单的语言解释，直到写不下去为止。写不下去的地方，就是他还没真正理解的地方。

这个方法被后人整理成"费曼技巧"，但我觉得它的核心其实很朴素：**写作是一种严苛的测试**。

## 思考的幻觉

我们在脑子里"想"一件事的时候，很容易产生一种理解的幻觉。思路看起来流畅，逻辑好像连贯，结论似乎自然而然就到了。但一旦要把它写出来，就会发现有个地方跳过去了，有个概念模糊了，有个"因此"其实并不成立。

这不是语言表达的问题，而是思维本身的问题。

写作迫使你把内部的模糊状态具体化。词语是有边界的，句子是有结构的，段落是有逻辑的。你必须做出选择，而选择会暴露你的判断。

## 不是记录，而是生产

有一种常见的误解：写作是把脑子里已有的想法记录下来。

但更多时候，想法是在写作的过程中才出现的。你打下第一句话，第二句话才变得可能；第一段写完，你才知道下一段应该去哪里。写作不是思考的下游，它参与了思考本身。

```go
// 这段代码的注释写得比代码本身花了更长时间
// 因为要把"为什么这样写"解释清楚，
// 才发现自己对"为什么"的理解并不充分。
func process(data []byte) (Result, error) {
    // ...
}
```

程序员应该对这种体验很熟悉：给代码写注释的时候，才发现代码本身有问题。

## 写给未来的自己

还有一个我慢慢意识到的用处：写下来的东西是可以检验的。

三个月后翻出这篇文章，你会知道当时的自己在想什么，相信什么，错在哪里。没有写下来的想法会被记忆悄悄篡改——我们总是倾向于记住"后来证明是对的"那个版本的自己。

写作留住的，是那个还不确定的、正在摸索的自己。那才是真实发生过的事。
